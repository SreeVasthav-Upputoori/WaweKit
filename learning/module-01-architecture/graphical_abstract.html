Wawekit · Module 01

Project Architecture — the Walking Skeleton

A launchable, empty-but-professional desktop shell with the infrastructure every future module reuses.

The Problem & The Approach

Problem
  • A 20-module app built ad-hoc rots into an unmaintainable single-file script.
  • Every module would otherwise re-invent logging, config, paths, and window plumbing.
  • Science tangled with UI is impossible to test or publish from.
Approach — Layered "walking skeleton"
  • Ship an end-to-end runnable app with zero features first.
  • Enforce a strict downward dependency rule between layers.
  • Build shared infrastructure once; grow features into it.

Layered Architecture (dependencies point downward only)

gui  — PySide6 windows & widgets  MainWindow, ThemeManager, QSS themes
▼ may import
services  — orchestration & background workers  (grows Module 2+)
▼ may import
models  — RDKit-backed domain objects  no Qt (grows Module 2+)
▼ may import
core  — constants · paths · config · logging  imports nothing above it

Startup Sequence (composition root: app.run())

1QApplication + identity
2Ensure OS dirs
3Load config (TOML)
4Init logging
5Apply theme
6Show MainWindow
7Run event loop

What Was Built

Infrastructure (core)
  • constants — app identity in one place
  • paths — cross-platform dirs via QStandardPaths
  • config — frozen AppConfig dataclass + layered TOML
  • logging — console + rotating file, idempotent
Desktop shell (gui)
  • MainWindow — menus, toolbar, status bar, dock
  • Shared QActions — one action drives menu + toolbar
  • ThemeManager — runtime dark/light via QSS
  • About dialog, DI of config + theme manager

Key Engineering Decisions
