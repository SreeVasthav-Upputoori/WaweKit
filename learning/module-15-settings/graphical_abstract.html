Settings — Preferences That Survive a Restart

Module 1 built AppConfig and only read it. Module 15 adds the other half: writing — and remembering the window.

Two backends, each for what it's good at

Human prefs → TOML

  • theme · log level · remember-layout flag
  • <config_dir>/settings.toml — hand-editable
  • Hand-rolled 12-line serializer (flat str/int/bool) → no TOML-writer dep
  • bool before int: True→true, not 1 (tested)

Window state → QSettings

  • geometry + dock layout = opaque QByteArray
  • Qt-native (registry) — not for hand-editing
  • restoreState works: every dock/toolbar has an objectName

When it saves

  • On close — capture the runtime theme + layout, save once. So Ctrl+T survives restart without a disk write per toggle.
  • On Settings OK — save + apply live (theme → ThemeManager repaints all panels; log level → setup_logging).
  • At startup — resize to default first, then overlay saved geometry if present (first run = sensible default; later = remembered).

Key Ideas

Clean design
  • Dialog is a pure editor — config in, config out; the window applies + persists
  • config(base) overlays only exposed fields → nothing silently reset
  • Restore Defaults button
Foundation reused
  • AppConfig (M1) · layered TOML load (M1) · ThemeManager signal (M3)
  • objectNames on docks (M3–14) make restoreState just work
  • Honest research note: this is plumbing, no novelty

Result

theme + log level + layout persist dependency-free TOML writer round-trip verified end to end verified: 263 tests · ruff · black · -W error
